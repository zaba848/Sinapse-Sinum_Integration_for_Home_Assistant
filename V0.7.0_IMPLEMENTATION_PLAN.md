## P4 Backlog: Priority Breakdown

### **P4.1 — IAQ/AQ Live Probe** ⭐ START HERE

**Priority**: HIGH (gatekeeper for feature implementation)  
**Effort**: 15 minutes  
**Hardware**: tablica-wtp (30 virtual, 254 WTP, 8 SBUS...)

**Objective**: Validate live WTP sensor payloads for:
- `iaq_sensor` (Indoor Air Quality)
- `aq_sensor` (Air Quality)
- `air_quality_sensor` (PM2.5/PM10)

**Current State** (from codebase):
```python
# descriptors/sensor.py: Already has field mappings
"iaq": ("IAQ", None, None),  # Index
"air_quality": ("Air Quality", None, "%"),
"pm25": ("PM2.5", "mdi:lung", "µg/m³"),
"pm10": ("PM10", "mdi:lung", "µg/m³"),
```

**What to Probe** (via tablica-wtp API):
```bash
# Get device family for WTP hub
curl -X GET http://10.0.61.132/api/v1/devices \
  -H "Authorization: Bearer $SINUM_WTP_TOKEN" \
  | jq '.[] | select(.type == "iaq_sensor")'

# Expected response structure:
{
  "id": "<device-id>",
  "name": "<device-name>",
  "type": "iaq_sensor",
  "state": {
    "iaq": <number>,
    "air_quality": <percentage>,
    "pm25": <number>,
    "pm10": <number>,
    "temperature": <float>,  # usually included
    "humidity": <float>       # usually included
  }
}
```

**Success Criteria**:
- ✅ Response fields match descriptor keys (`iaq`, `air_quality`, `pm25`, `pm10`)
- ✅ Values are numeric and in expected ranges (IAQ 0-500, PM 0-1000)
- ✅ Descriptors don't need changes (no new fields discovered)

**If fields differ**:
- Document new fields in hardware probe report
- Add to descriptors if needed
- Create follow-up task for P7 (v0.7.1)

**Deliverable**:
```markdown

### Device 2: Bedroom IAQ
- Fields: ✅ (same as Device 1)

### Device 3: Kitchen AQ
- Fields: ✅ (same as Device 1)

## Conclusion
Descriptors match live payloads. No changes needed to sensor.py.
Implementation can proceed.
```

**Timeline**:
- Prep credentials: 2 min
- Query API: 5 min

- **Total: 15 min**

**Next Step**: If probe succeeds → implement P5 features
